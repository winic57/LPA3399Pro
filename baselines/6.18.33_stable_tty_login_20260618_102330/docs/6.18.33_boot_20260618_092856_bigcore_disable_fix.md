# 6.18.33 boot_20260618_092856 分析与 SD 卡修改记录

> 时间：2026-06-18 20260618_093300

## 1. 本轮结论

上一轮屏蔽 `modprobe@drm/configfs/efi_pstore` 有效：日志中没有再启动这些服务，也没有在原位置 panic。

系统已经继续推进到 systemd 早期服务阶段，但在多个用户态进程触发 syscall/mount/exec 路径时出现内核随机内存破坏：

- `udevadm`：`invoke_syscall` Oops
- `(md-udevd)`：`make_visible/copy_tree/path_mount` Oops
- `setup.sh`：定时器中断/exec 路径 Oops，最终 `Fatal exception in interrupt`

关键特征：首次 Oops 均发生在 CPU4/CPU5。RK3399 CPU4/5 是 A72 big cores；当前使用的是 Rock Pi N10 主线 DTB 替代 LPA3399Pro vendor DTB，OPP/电源/电压/时钟不完全匹配时，大核在 systemd 并发负载下出现随机内核内存破坏是高概率路径。

## 2. 本次 SD 卡修改

### 2.1 启动参数禁用 big cores

已修改 boot 分区 `extlinux.conf`，追加：

```text
maxcpus=4
```

目的：只启用 CPU0-3 四个 A53 小核，避免 CPU4/5 A72 参与启动。

备份文件：

```text
/extlinux/extlinux.conf.before_maxcpus4_20260618_093300
```

### 2.2 降低 systemd 早期并发/负载

已临时 mask 以下靠近 Oops 的非关键服务：

- `systemd-udev-trigger.service`
- `systemd-resolved.service`
- `systemd-random-seed.service`
- `armbian-zram-config.service`
- `plymouth-read-write.service`
- `console-setup.service`
- `keyboard-setup.service`

保留 `systemd-udevd.service` 本体，避免 /dev 基础设备管理完全缺失。

## 3. 预期

下一轮日志应看到：

- CPU4/CPU5 不再 online 或不参与调度
- 不再出现 CPU4/CPU5 上的随机 Oops
- systemd 推进到 `serial-getty@ttyS2` / root autologin

若仍在小核上 Oops，则说明不是 big core OPP/电源问题，而是更底层的内存映射/DT reserved-memory/内核 6.18.33 bug，需要转向：

1. 启动参数改成 `nosmp` 单核验证；或
2. 回退到更稳定的 6.1.y 主线内核；或
3. 继续精简 DTB reserved-memory / DMA/IOMMU 节点。
