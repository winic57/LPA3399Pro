## 1. 现象

最新日志不再卡在早期 DTB/GMAC/PCIe 阶段，而是进入 rootfs 挂载阶段后 panic：

```text
Disabling rootwait; root= is invalid.
VFS: Cannot open root device "UUID=d52f7f8b-fa0e-4c9f-a552-d6df4329ac87" or unknown-block(0,0): error -6
Kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block(0,0)
```

## 2. 关键证据

日志已经证明 SD 卡枚举成功：

```text
mmcblk1: mmc1:0001 SD16G 14.4 GiB
mmcblk1: p1 p2
```

内核列出的可用分区中，SD 卡 root 分区为：

```text
b362 3070976 mmcblk1p2 61ec8aeb-3d1a-48fa-a9da-54d744ed8bdf
```

这里显示的是 GPT PARTUUID；当前 extlinux 只加载 Image 和 DTB，未加载 initramfs，因此早期内核不能解析 `root=UUID=...` 这种 filesystem UUID 写法，应改用 `root=PARTUUID=...`。

## 3. 本次已直接修改 SD 卡

已挂载 SD 卡 boot 分区 `/dev/sdc1`，修改：

```diff
- root=UUID=d52f7f8b-fa0e-4c9f-a552-d6df4329ac87
+ root=PARTUUID=61ec8aeb-3d1a-48fa-a9da-54d744ed8bdf
```

备份文件：

```text
/boot/extlinux/extlinux.conf.before_partuuid_20260618_085921
```

修改后的关键行：

```text
见 SD 卡 /extlinux/extlinux.conf：root=PARTUUID=61ec8aeb-3d1a-48fa-a9da-54d744ed8bdf
```

## 4. 保持不变的策略

- `/ethernet@fe300000` 继续保持 `disabled`：GMAC DMA SWR 位硬件卡死，主线 stmmac probe 会触发死机。
- `/pcie@f8000000` 继续保持 `okay`：PCIe link training 虽然 `-110` 超时，但可优雅失败，并且对 MMC 探测时序有正面作用。
- DRM/VOP/watchdog 继续保持 disabled，先保证最小启动闭环。
