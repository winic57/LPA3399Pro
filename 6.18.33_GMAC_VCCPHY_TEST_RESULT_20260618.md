# 6.18.33 GMAC vcc_phy 测试结果（2026-06-18）

## 1. 目的

验证当前主线/Armbian DTB 中 `phy-supply = <&vccio_3v3>` 是否是导致 GMAC DMA reset failed 的关键差异。

参考 vendor DTS：

- `LPA3399Pro-SDK-Linux-V3.0/kernel/arch/arm64/boot/dts/rockchip/rk3399pro-neardi-linux-lc110-base.dts`
- `LPA3399Pro-SDK-Linux-V3.0/kernel/arch/arm64/boot/dts/rockchip/rk3399pro-neardi-linux-ld110-phy.dtsi`

vendor 期望：

```dts
phy-supply = <&vcc_phy>;
```

而当前测试前 DTB 中 `phy-supply` 实际指向 `vccio_3v3`（phandle `0x22`）。

## 2. 变更

基于：

```text
/mnt/sdboot/dtb/rockchip/gmac-neardi-vendor-delay-test.dtb
```

创建：

```text
/mnt/sdboot/dtb/rockchip/gmac-vccphy-test.dtb
```

修改内容：

1. 新增固定 regulator 节点：

```dts
vcc-phy-regulator {
    compatible = "regulator-fixed";
    regulator-name = "vcc_phy";
    regulator-always-on;
    regulator-boot-on;
};
```

2. 将：

```dts
/ethernet@fe300000/phy-supply
```

从 `vccio_3v3` 改为指向新增 `vcc_phy` 节点。

3. 其余保持 vendor neardi 参数：

```dts
clock_in_out = "input"
snps,reset-gpio = <gpio3 PB7 GPIO_ACTIVE_LOW>
tx_delay = <0x21>
rx_delay = <0x15>
```

## 3. 验证到的运行态状态

运行态 DT 已确认：

- `/proc/device-tree/vcc-phy-regulator/regulator-name = "vcc_phy"`
- `/proc/device-tree/ethernet@fe300000/phy-supply = <0xc1>`

regulator 摘要：

```text
vcc_phy                          2    1      0 unknown
   fe300000.ethernet-phy         1
```

说明新 `vcc_phy` 节点已经被内核消费，`phy-supply` 指向修正生效。

## 4. 启动与 dmesg 结果

仍然可以稳定进入系统，GMAC probe 正常：

```text
rk_gmac-dwmac fe300000.ethernet: clock input or output? (input).
rk_gmac-dwmac fe300000.ethernet: TX delay(0x21).
rk_gmac-dwmac fe300000.ethernet: RX delay(0x15).
rk_gmac-dwmac fe300000.ethernet: init for RGMII
YT8521 Gigabit Ethernet stmmac-0:00: attached PHY driver
YT8521 Gigabit Ethernet stmmac-0:03: attached PHY driver
```

但 `eth0` open 仍失败：

```text
rk_gmac-dwmac fe300000.ethernet eth0: Failed to reset the dma
rk_gmac-dwmac fe300000.ethernet eth0: stmmac_hw_setup: DMA engine initialization failed
rk_gmac-dwmac fe300000.ethernet eth0: __stmmac_open: Hw setup failed
```

## 5. 结论

**结论：单独补齐 vendor 风格 `vcc_phy/phy-supply` 并不能解决 GMAC DMA reset failed。**

这说明：

- `phy-supply` 的确是一个 board-level 差异；
- 但它不是当前 DMA reset failed 的决定性根因。

## 6. 推进价值

本轮仍然有两点重要收获：

1. 排除了“仅仅是 PHY 电源 regulator 语义错误”这一主嫌疑；
2. 证明补齐 `vcc_phy` 后系统仍稳定，不会引入新的死机/卡启动问题。

## 7. 下一步建议

下一步应转向更高价值方向：

1. **显式 PHY 子节点 + `phy-handle` 测试**
   - 限制只绑定 `stmmac-0:00`
   - 避免当前默认 MDIO 扫描挂出两个 PHY 地址（`0x00` / `0x03`）
   - 增加 YT8521 相关属性（如 keep PLL / 禁止 auto sleep）

2. **一致的 output 模式测试**
   - 不仅改 `clock_in_out=output`
   - 还要移除/重做 `assigned-clocks` / `assigned-clock-parents`
   - 避免此前“output 模式 + clkin_gmac 仍作父源”的自相矛盾配置

3. **若上述仍失败，进入驱动侧补丁**
   - 对比 vendor 4.4 `dwmac-rk.c` 与当前 6.18 `dwmac-rockchip`
   - 重点看 RK3399 GRF/clock/reset/init 顺序
