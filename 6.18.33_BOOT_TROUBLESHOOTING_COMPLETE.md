## 技术细节

### idbloader 问题分析

**问题**: ophub 默认的 idbloader 无法被 RK3399 BootROM 识别

**原因**: 
- ophub 生成的 idbloader 缺少正确的 rksd 头
- BootROM 检测 SD 卡 sector 64 时无法识别，跳过 SD 卡
- 转而从 eMMC 启动

**解决方案**:
- 使用 SDK 版本的 idbloader（`make.sh --idblock` 生成）
- 写入到 SD 卡 sector 64（偏移 32KB）
- BootROM 可以正确识别并启动

**参考**: `LPA3399Pro_Auto_SD_Boot_Guide_20260614.md`

### boot.scr 问题分析

**问题**: boot.scr 脚本执行时显示乱码

**最初猜测**: 文件损坏或编译错误

**验证过程**:
1. 检查 boot.cmd 源文件 → ✅ 正常
2. 重新编译 boot.scr → ✅ 格式正确，仍失败
3. 使用原始镜像的 boot.scr → ✅ 完全一致，仍失败

**真实原因**: U-Boot 环境变量损坏

**证据**:
```
Line 158: *** Warning - bad CRC, using default environment
```

这表明：
- U-Boot 的环境分区（env partition）CRC 校验失败
- U-Boot 回退到硬编码的默认环境
- 默认环境中的变量定义可能与 Armbian boot.scr 不兼容

### U-Boot 环境变量损坏原因

可能原因：
1. **eMMC 环境分区损坏** - U-Boot 从 eMMC 加载（vendor U-Boot）
2. **从未保存过正确的环境** - 第一次使用 SD 卡启动
3. **版本不兼容** - vendor U-Boot vs Armbian 期望的环境

**关键点**: 即使从 SD 卡启动，U-Boot 本身仍来自 eMMC（sector 0x4000），其环境变量也保存在 eMMC。

## 解决方案

### 方案 A: 手动启动（立即可用）⭐

绕过 boot.scr，直接在 U-Boot 命令行手动加载内核。

**步骤**（在 `=>` 提示符下执行）:

```bash
# 方法 1: 使用环境变量（优先尝试）
mmc dev 1
mmc rescan
load mmc 1:1 ${kernel_addr_r} /Image
load mmc 1:1 ${ramdisk_addr_r} /uInitrd
load mmc 1:1 ${fdt_addr_r} /dtb/rockchip/rk3399pro-neardi-linux-lc110-base.dtb
setenv bootargs "root=/dev/mmcblk1p2 rootwait rootfstype=ext4 console=ttyS2,1500000 loglevel=7"
booti ${kernel_addr_r} ${ramdisk_addr_r} ${fdt_addr_r}
```

如果方法 1 失败（变量未定义），使用固定地址：

```bash
# 方法 2: 固定地址
mmc dev 1
mmc rescan
load mmc 1:1 0x02080000 /Image
load mmc 1:1 0x06000000 /uInitrd
load mmc 1:1 0x01f00000 /dtb/rockchip/rk3399pro-neardi-linux-lc110-base.dtb
setenv bootargs "root=/dev/mmcblk1p2 rootwait rootfstype=ext4 console=ttyS2,1500000 loglevel=7"
booti 0x02080000 0x06000000 0x01f00000
```

### 方案 B: 修复 U-Boot 环境（持久化）

如果方案 A 成功启动，可以持久化配置：

```bash
# 在 U-Boot 命令行
setenv bootcmd 'mmc dev 1; mmc rescan; load mmc 1:1 0x02080000 /Image; load mmc 1:1 0x06000000 /uInitrd; load mmc 1:1 0x01f00000 /dtb/rockchip/rk3399pro-neardi-linux-lc110-base.dtb; setenv bootargs "root=/dev/mmcblk1p2 rootwait rootfstype=ext4 console=ttyS2,1500000 loglevel=7"; booti 0x02080000 0x06000000 0x01f00000'
saveenv
reset
```

### 方案 C: 使用 extlinux.conf（备用）

```bash
# 在 U-Boot 命令行
setenv boot_scripts ""
setenv scan_dev_for_extlinux "if test -e ${devtype} ${devnum}:${distro_bootpart} /extlinux/extlinux.conf; then echo Found extlinux config...; sysboot ${devtype} ${devnum}:${distro_bootpart} any ${scriptaddr} /extlinux/extlinux.conf; fi"
run scan_dev_for_extlinux
```

## 内存地址说明

### RK3399 标准地址映射

| 组件 | 变量名 | 标准地址 | 备用地址 | 说明 |
|---|---|---|---|---|
| Kernel | ${kernel_addr_r} | 0x02080000 | 0x00280000 | ARM64 Image |
| Initrd | ${ramdisk_addr_r} | 0x06000000 | 0x0a200000 | uInitrd |
| DTB | ${fdt_addr_r} | 0x01f00000 | 0x08300000 | Device Tree |

**地址选择原则**:
- 优先使用环境变量（如果定义）
- 回退到标准地址 (0x02080000 系列)
- 最后尝试备用地址 (0x00280000 系列)
