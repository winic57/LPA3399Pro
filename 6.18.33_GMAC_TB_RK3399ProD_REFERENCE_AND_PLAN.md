## 1. 结论摘要

当前 6.18.33 内核中的有线网卡问题，**不是因为 `dwmac-rockchip` 驱动缺失**，而是 DTB/板级配置导致 GMAC 没有被正常启用，且历史测试表明 GMAC 相关探测/打开阶段可能触发死机。

已确认：
- 内核已编入 `CONFIG_DWMAC_ROCKCHIP=y`
- `/sys/bus/platform/drivers/rk_gmac-dwmac` 存在
- 当前稳定 DTB 中 `/ethernet@fe300000 status = "disabled"`
- 因此 `eth0` 不出现是预期结果


## 1.1 是否有可直接使用的 DTB

本次在 TB-RK3399ProD 仓库内未发现适合当前 SD 启动环境可直接替换使用的已编译 `.dtb` 文件。仓库提供的主要价值是：

- 4.4.194 vendor DTS/DTSI 参考；
- 6.x ophub 适配 patch；
- GMAC reset GPIO、delay、clock、phy-supply 等板级配置参考。

因此建议不要直接替换当前稳定 DTB，而是按参考值制作独立 `gmac-test.dtb` 逐项验证。

## 2. TB-RK3399ProD 参考中可用的 GMAC 资料

### 2.1 文档参考

`docs/设备树适配/设备树适配.md` 给出的 GMAC 建议配置：

```dts
&gmac {
    phy-supply = <&vcc_phy>;
    phy-mode = "rgmii";
    clock_in_out = "input";
    snps,reset-gpio = <&gpio3 RK_PC0 GPIO_ACTIVE_LOW>;
    snps,reset-active-low;
    snps,reset-delays-us = <0 10000 50000>;
    assigned-clocks = <&cru SCLK_RMII_SRC>;
    assigned-clock-parents = <&clkin_gmac>;
    pinctrl-names = "default";
    pinctrl-0 = <&rgmii_pins>;
    tx_delay = <0x28>;
    rx_delay = <0x20>;
    status = "okay";
};
```

并明确写了：`CONFIG_DWMAC_ROCKCHIP=y`，否则缺失驱动。

### 2.2 4.4.194 参考 DTS

- `kernel-4.4.194-official/dts/rk3399pro-toybrick-prod.dtsi`

  - `rx_delay = <0x20>`

- `kernel-4.4.194-official/dts/rk3399pro-toybrick-96ai.dtsi`

- `kernel-4.4.194-official/dts/rk3399pro-evb-v11-linux.dts`
  - `snps,reset-gpio = <&gpio3 RK_PB7 GPIO_ACTIVE_LOW>`

### 2.3 6.x 参考补丁

`ophub_6.6.y/0001-add-toybrick-rk3399prod-support-for-kernel-6.patch` 中，Toybrick prod 的 GMAC 仍是：

- `snps,reset-gpio = <&gpio3 RK_PC0 GPIO_ACTIVE_LOW>`
- `rx_delay = <0x20>`
- `status = "okay"`

## 3. 与当前基线 DTB 的差异

当前稳定基线 DTB 关键值：

- `/ethernet@fe300000 status = disabled`
- `compatible = "rockchip,rk3399-gmac"`
- `phy-mode = "rgmii"`
- `clock_in_out = "input"`
- `snps,reset-gpio = <&gpio3 RK_PB7 GPIO_ACTIVE_LOW>`
- `tx_delay = <0x28>`
- `rx_delay = <0x11>`

对比 TB-RK3399ProD 参考：

- **status**: 当前 disabled，参考应为 okay
- **reset GPIO**: 当前 PB7，参考 prod/96ai 以 PC0 为主
- **rx_delay**: 当前 0x11，参考 prod 为 0x20，96ai/EVB 为 0x11

## 4. 解释与判断

### 4.1 驱动不是根因

当前现象不是“内核没编 GMAC 驱动”，而是：

1. 驱动已存在；
2. DTB 节点被禁用；
3. 之前的实验还显示 GMAC 探测/启动阶段可能会把板子打挂。

### 4.2 最关键的不确定项

**PB7 vs PC0** 这件事不能直接按参考仓库盲改。

- TB-RK3399ProD 的 prod/96ai 文档强烈指向 `gpio3 PC0`
- 但当前基线 DTB 是 `gpio3 PB7`
- 不同板型可能硬件真实连线不同

因此建议按“最小改动、分步实验”处理。

## 5. 建议的安全试验顺序

### 方案 A：只把 GMAC 从 disabled 改为 okay

先不动 reset GPIO、delay、clock 配置，只改：

```dts
/ethernet@fe300000/status = "okay"
```

目的：验证是否能枚举出 eth0，确认 probe 行为。

### 方案 B：若 A 能启动但网卡异常，再调 delay

优先试：
- `rx_delay = <0x20>`
- 保持 `tx_delay = <0x28>` 不变

### 方案 C：再考虑 reset GPIO

仅当确认当前板子与 TB-RK3399ProD prod 同源连线时，才考虑把：

- `snps,reset-gpio` 从 PB7 改成 PC0

### 方案 D：如果 probe/open 仍然硬死机

回到驱动侧排查：
- `dwmac-rk.c`
- PHY reset / clock / GRF 逻辑
- DMA reset 触发点

## 6. 建议的实验方式

为避免破坏当前稳定基线，建议：

1. 复制当前 DTB 为 `gmac-test.dtb`
2. 只做一处修改
3. 用独立 extlinux 启动项测试
4. 观察 TTL：
   ```bash
   dmesg | grep -iE 'gmac|stmmac|dwmac|eth0|phy|motorcomm|yt8521'
   ip link
   ```
5. 如 eth0 出现，再手工 `ip link set eth0 up` 观察是否触发错误

## 7. 当前推荐结论

- **可用参考是存在的**，TB-RK3399ProD 的文档和 6.x patch 都能作为 GMAC DTS 参考
- 其中最值得优先试验的改动是：`status = "okay"`
- **不要直接把当前稳定基线改成 prod 参考全量配置**
- 当前更像是“DTB 禁用 + 板级 GMAC 兼容性/时序问题”，不是缺驱动
