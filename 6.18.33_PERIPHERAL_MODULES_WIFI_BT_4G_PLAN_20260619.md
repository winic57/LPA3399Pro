## 2. 阶段 A：完整安装 6.18.33 modules

### A1. 确认内核源码、配置和版本一致性

在构建机侧确认：

```bash
uname -r                           # 板端应为 6.18.33
make kernelrelease                  # 构建树应输出 6.18.33
zcat /proc/config.gz > board.config # 板端配置备份，如可用
```

重点确认 `CONFIG_LOCALVERSION`、编译器、架构、DTB 与已启动 Image 一致，避免装错模块版本。

### A2. 建议至少打开/确认以下内核选项

WiFi/通用无线：

```text
CONFIG_RFKILL=m 或 y
CONFIG_CFG80211=m
CONFIG_MAC80211=m
CONFIG_WIRELESS=y
CONFIG_BRCMUTIL=m
CONFIG_BRCMFMAC=m
CONFIG_BRCMFMAC_SDIO=y
CONFIG_BRCMFMAC_USB=y    # 若存在 USB WiFi/BT combo 或测试 USB WiFi
```

Bluetooth：

```text
CONFIG_BT=m 或 y
CONFIG_BT_BCM=m
CONFIG_BT_HCIBTUSB=m     # USB BT 场景
CONFIG_BT_HCIUART=m 或 y # UART BT 场景
CONFIG_BT_HCIUART_BCM=y  # Broadcom UART BT 场景
CONFIG_BT_HCIUART_SERDEV=y # 若 DT 用 serdev bluetooth 子节点
```

USB/4G/串口/网络：

```text
CONFIG_USB_SERIAL=m
CONFIG_USB_SERIAL_OPTION=m
CONFIG_USB_SERIAL_WWAN=m
CONFIG_USB_NET_QMI_WWAN=m
CONFIG_USB_NET_CDC_MBIM=m
CONFIG_USB_NET_CDC_NCM=m
CONFIG_USB_NET_CDC_ETHER=m
CONFIG_PPP=m
CONFIG_PPP_ASYNC=m
CONFIG_PPP_DEFLATE=m
```

调试辅助：

```text
CONFIG_DEVTMPFS=y
CONFIG_FW_LOADER=y
CONFIG_DYNAMIC_DEBUG=y 可选
```

### A3. 构建并安装 modules

构建机侧示例：

```bash
make -j$(nproc) Image modules dtbs
make modules_install INSTALL_MOD_PATH=/tmp/rk3399pro-6.18.33-modules
```

生成包：

```bash
cd /tmp/rk3399pro-6.18.33-modules
sudo tar --numeric-owner -czf /mnt/sdb3/LPA3399Pro/build_artifacts/modules-6.18.33-$(date +%Y%m%d_%H%M%S).tar.gz lib/modules/6.18.33
```

部署到板端 rootfs 后执行：

```bash
depmod -a 6.18.33
find /lib/modules/6.18.33 -maxdepth 2 -type f | head
modprobe rfkill
modprobe cfg80211
modprobe brcmfmac
modprobe bluetooth || true
modprobe hci_uart || true
modprobe option qmi_wwan cdc_mbim || true
```

### A4. modules 验收标准

通过条件：

```text
/lib/modules/6.18.33 存在
modules.dep/modules.alias 存在
modprobe rfkill/cfg80211/brcmfmac 不再报 FATAL module not found
/dev/rfkill 出现，或 rfkill 子系统可枚举
lsmod 能看到已加载无线/BT/4G相关模块
```

失败处理：

- 若 `invalid module format`：检查 `vermagic`、`CONFIG_LOCALVERSION`、SMP/PREEMPT、编译器版本。
- 若 `module not found`：检查安装路径是否为准确的 `/lib/modules/6.18.33`。
- 若模块加载但设备未出现：进入 B/C/D/E 阶段继续查 firmware/DT/电源/时钟。

## 3. 阶段 B：补齐 WiFi firmware 与 NVRAM

### B1. 识别板载 WiFi 芯片

优先从以下位置确认芯片型号：

```bash
dmesg | grep -iE 'brcm|wifi|wlan|sdio|mmc'
find /proc/device-tree -iname '*wifi*' -o -iname '*sdio*' -o -iname '*wireless*'
strings /proc/device-tree/* 2>/dev/null | grep -iE 'ap6|brcm|bcm|wifi' || true
```

若是 AP6255/AP6256/AP6356S 等 Broadcom SDIO combo，需要匹配 brcmfmac firmware + board NVRAM。

### B2. 固件/NVRAM 放置建议

常见路径：

```text
/lib/firmware/brcm/brcmfmac*.bin
/lib/firmware/brcm/brcmfmac*.txt
/lib/firmware/brcm/brcmfmac*.clm_blob
```

根据 dmesg 中 brcmfmac 请求的文件名补齐，避免盲放错型号 NVRAM。

### B3. WiFi bring-up 命令

```bash
modprobe rfkill
modprobe cfg80211
modprobe brcmfmac
dmesg | grep -iE 'brcm|firmware|sdio|wlan|cfg80211' | tail -120
rfkill list
ip link show
iw dev
```

### B4. WiFi 验收标准

通过条件：

```text
iw dev 能看到 wlan0 或类似接口
rfkill list 能看到 wlan 或 phy
brcmfmac 无 firmware/NVRAM 加载失败
可扫描 AP：iw dev wlan0 scan
可 DHCP 获取 WiFi 地址
ping 网关 300 包 0% loss
```

## 4. 阶段 C：补齐 Bluetooth firmware / NVRAM / 工具

### C1. 确认 BT 总线类型

优先判断是 USB BT 还是 UART BT：

```bash
lsusb
find /proc/device-tree -iname '*bluetooth*' -o -iname '*uart*'
dmesg | grep -iE 'bluetooth|btbcm|hci|uart|ttyS|ttyFIQ'
```

判断逻辑：

- 若 `lsusb` 出现 BT 设备：偏 USB BT，重点查 `btusb/btbcm`。
- 若 DT 有 `bluetooth` 子节点挂在 UART：偏 UART BT，重点查 `hci_uart/btbcm/serdev` 和 UART pinctrl/clock/reset。

### C2. 用户态工具

网络/DNS 修复后建议安装：

```bash
apt update
apt install -y bluez rfkill wireless-tools iw network-manager modemmanager usbutils pciutils
```

若 apt 暂不可用，至少先保证 kernel modules + firmware，用户态工具后补。

### C3. BT bring-up 命令

USB BT：

```bash
modprobe bluetooth
modprobe btusb
modprobe btbcm
hciconfig -a || bluetoothctl list || btmgmt info
```

UART Broadcom BT：

```bash
modprobe bluetooth
modprobe hci_uart
modprobe btbcm
# 若非 serdev 自动绑定，需要用 btattach/brcm_patchram_plus 等方式绑定具体 tty
```

### C4. BT 验收标准

通过条件：

```text
/sys/class/bluetooth/hci0 出现
bluetoothctl list 能看到 controller
rfkill list 能看到 bluetooth
bluetoothctl scan on 能发现周边设备
```

## 5. 阶段 D：板载 4G 模块 bring-up 计划

### D1. 识别 4G 模块连接方式

常见方式：USB 2.0/USB 3.0、PCIe、UART 控制口、GPIO 电源/复位。先做枚举：

```bash
lsusb -t
lsusb
lspci -nn || true
ip link
ls -l /dev/ttyUSB* /dev/cdc-wdm* /dev/wwan* 2>/dev/null || true
dmesg | grep -iE 'wwan|qmi|mbim|cdc|option|usbserial|modem|lte|4g|sim|ttyUSB|cdc-wdm|pcie' | tail -160
```

### D2. 4G 内核模块

按模块类型加载：

```bash
modprobe usbserial
modprobe option
modprobe qmi_wwan
modprobe cdc_mbim
modprobe cdc_ncm
modprobe cdc_ether
modprobe ppp_async
```

### D3. 4G firmware / NVRAM / 配置

4G 模块通常需要：

- 模块自身 firmware：多数在 4G 模块内部，不一定放 Linux rootfs。
- 运营商 APN 配置。
- USB composition 模式确认：QMI / MBIM / ECM / RNDIS / PPP。
- 若板子 DT 控制 4G 电源/复位/使能 GPIO，需要确认 GPIO 默认态和上电时序。

建议记录：

```text
模块厂商/型号
USB VID:PID 或 PCI ID
AT 口 ttyUSB 编号
数据口类型：cdc-wdm/QMI、MBIM、ECM、PPP
SIM 卡状态
APN
```

### D4. 4G 用户态工具

推荐：

```bash
apt install -y modemmanager libqmi-utils libmbim-utils ppp minicom picocom
```

验证：

```bash
mmcli -L
mmcli -m 0
qmicli -d /dev/cdc-wdm0 --dms-get-operating-mode || true
mbimcli -d /dev/cdc-wdm0 --query-device-caps || true
```

### D5. 4G 验收标准

通过条件：

```text
lsusb/lspci 能看到 4G 模块
/dev/ttyUSB* 或 /dev/cdc-wdm0 出现
ModemManager 能识别 modem
SIM 状态正常
能注册运营商网络
能获取移动网络 IP
ping 运营商网关或 8.8.8.8 成功
```

## 6. 阶段 E：复查 SDIO pwrseq 外部时钟

当前关键错误：

```text
platform sdio-pwrseq: deferred probe pending: pwrseq_simple: external clock not ready
```

### E1. 复查 DT 节点

```bash
find /proc/device-tree -path '*sdio*' -o -path '*pwrseq*' -o -path '*wifi*'
for n in /proc/device-tree/sdio-pwrseq/*; do echo "--- $n"; hexdump -C "$n" 2>/dev/null || true; done
```

重点检查：

```text
sdio-pwrseq compatible = "mmc-pwrseq-simple"
ext_clock / clocks / clock-names 是否指向有效 32K clock
reset-gpios / power-gpios 极性是否正确
mmc@fe310000 status = "okay"
cap-sdio-irq / keep-power-in-suspend / non-removable 是否存在
pinctrl 是否包含 WiFi enable/reset/host-wake
```

### E2. 复查 clock provider

```bash
cat /sys/kernel/debug/clk/clk_summary 2>/dev/null | grep -iE '32k|xin|wifi|rtc|clkout|rk808|rk805|pmu' || true
dmesg | grep -iE 'clk|clock|rk808|rk805|pmic|32k|pwrseq|sdio' | tail -160
```

重点判断：

- WiFi 需要的 32.768K 外部时钟是否由 PMIC、RTC、SoC clock out 或固定时钟提供。
- DT 中 pwrseq 引用的 clock 是否存在、名称是否正确。
- 若 vendor DTS 使用固定时钟节点，应把对应 fixed-clock / clock-output-names 补进 mainline DT。

### E3. 修复方向

优先级：

1. 对照厂商 DTS，恢复 WiFi 32K clock provider。
2. 修正 `sdio-pwrseq` 的 `clocks` / `clock-names` / reset GPIO。
3. 确认 SDIO 控制器 `mmc@fe310000` 的 pinctrl、供电 regulator、bus-width、non-removable。
4. modules 和 firmware 补齐后再判断是否仍 deferred probe。

### E4. SDIO/WiFi 验收标准

通过条件：

```text
dmesg 不再出现 sdio-pwrseq external clock not ready
mmc/sdio 能枚举 WiFi function
brcmfmac 能加载 firmware/NVRAM
wlan0 出现
```

## 9. 风险与回滚

- 安装 modules 前备份原 rootfs 对应目录和 `/boot`。
- 若替换 Image/DTB，保留上一版可启动文件，并在 extlinux 中保留 fallback 菜单项。
- WiFi/BT/4G bring-up 前不要同时改多个 DT 节点；每次只改一个方向，保留 TTL 日志。
- 4G 涉及 SIM/APN/运营商网络，硬件枚举正常但拨号失败时，不要直接归因内核，需分层排查 SIM、APN、信号、模块制式和 USB composition。
