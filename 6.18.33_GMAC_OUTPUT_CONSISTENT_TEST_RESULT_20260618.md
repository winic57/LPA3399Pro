# 6.18.33 GMAC 一致 output 模式测试结果（2026-06-18）

## 1. 目的

修正上一轮 `clock_in_out=output` 但 `assigned-clock-parents` 仍指向 `clkin_gmac` 的“不一致配置”，做一次更干净的 output 模式测试。

## 2. 变更

基于：

```text
/mnt/sdboot/dtb/rockchip/gmac-phyhandle-pll-test.dtb
```

创建：

```text
/mnt/sdboot/dtb/rockchip/gmac-output-consistent-test.dtb
```

变更如下：

1. 改为：

```dts
clock_in_out = "output";
```

2. 删除：

```dts
assigned-clocks
assigned-clock-parents
```

以避免“MAC 配置为输出、但父时钟仍绑定外部 `clkin_gmac`”的逻辑冲突。

3. 保留前一轮已验证有效但不致命的改动：

- `vcc_phy`
- `phy-handle = <ethernet-phy@0>`
- `mdio/ethernet-phy@0`
- `motorcomm,keep-pll-enabled`
- `motorcomm,auto-sleep-disabled`

## 3. 运行态结果

运行态 DT：

```text
clock_in_out = output
phy-handle = <0xc2>
```

MDIO 仍只出现一个 PHY：

```text
stmmac-0:00
```

内核日志也明确进入 output 路径：

```text
rk_gmac-dwmac fe300000.ethernet: clock input or output? (output).
```

## 4. 关键时钟观测

在 output 测试下，`clk_summary` 仍显示：

```text
clkin_gmac      125000000 enabled
clk_rmii_src    125000000 enabled
clk_rmii_tx     125000000 enabled
clk_mac_ref     125000000 disabled
clk_mac_refout  125000000 disabled
clk_gmac         50000000 enabled
aclk_gmac       200000000 enabled
pclk_gmac        50000000 enabled
```

这表明即使 DT 语义上改成了 output，运行时关键参考时钟路径仍然没有表现出明显变化，`clk_mac_refout` 也没有真正被启用。

## 5. dmesg 结果

probe/attach 依旧正常：

```text
rk_gmac-dwmac fe300000.ethernet: clock input or output? (output).
rk_gmac-dwmac fe300000.ethernet: TX delay(0x21).
rk_gmac-dwmac fe300000.ethernet: RX delay(0x15).
rk_gmac-dwmac fe300000.ethernet: init for RGMII
rk_gmac-dwmac fe300000.ethernet eth0: PHY [stmmac-0:00] driver [YT8521 Gigabit Ethernet] (irq=POLL)
```

但 open 阶段仍完全同样失败：

```text
rk_gmac-dwmac fe300000.ethernet eth0: Failed to reset the dma
rk_gmac-dwmac fe300000.ethernet eth0: stmmac_hw_setup: DMA engine initialization failed
rk_gmac-dwmac fe300000.ethernet eth0: __stmmac_open: Hw setup failed
```

## 6. 结论

**结论：即便修正为更一致的 output 模式，DMA reset failed 依旧存在。**

这意味着：

1. “input/output 模式配置不一致”不是当前问题的主根因；
2. 仅靠 DTB 级别切换时钟方向，无法让 DMA reset 成功；
3. 问题更像是：
   - 主线 6.18 `dwmac-rockchip` 对 RK3399Pro 的 GRF/clock/reset 初始化顺序与 vendor 4.4 不一致；或
   - 某个 MAC 侧参考时钟/复位在 runtime 并未真正切换到硬件需要的状态。

## 7. 额外关键证据

在 `ethtool -d eth0` 的寄存器转储中，`ifup` 前后 **DMA 寄存器几乎完全不变**：

```text
DMA Reg0  = 0x00020101
DMA Reg10 = 0x00110001
DMA Reg22 = 0x000D0F17
```

说明 `ip link set eth0 up` 后，DMA reset 甚至没有推动控制器进入新的可见状态，进一步支持“驱动初始化序列/clock/reset 生效性异常”的判断。

## 8. 当前建议

不建议继续盲调 delay/reset GPIO/clock_in_out。

更高价值的下一步：

1. 对比 vendor 4.4 `dwmac-rk.c` 与当前 6.18 `dwmac-rockchip` 的 RK3399 流程；
2. 重点检查：
   - RK3399 `set_to_rgmii()` 对 GRF 的写入；
   - `gmac_clk_enable()` / 参考时钟选择；
   - reset assert/deassert 时序；
   - DMA soft reset 前是否缺少某个前置步骤；
3. 如果能拿到当前 6.18 对应源码，优先尝试最小补丁而不是继续堆 DTB 试验。
