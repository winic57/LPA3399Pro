# 6.18.33 有线网卡（GMAC）后续修改方案（供后续 agent 接手）

日期：2026-06-18  
目标：在 **保持 6.18.33 可稳定启动** 的前提下，继续推进 RK3399Pro 板载 GMAC 有线网卡可用。

---

## 1. 当前结论（先看这个）

### 1.1 已确认不是“驱动缺失”

当前内核已包含 GMAC 相关驱动：

- `CONFIG_STMMAC_ETH=y`
- `CONFIG_STMMAC_PLATFORM=y`
- `CONFIG_DWMAC_ROCKCHIP=y`

所以问题 **不是** `dwmac-rockchip` 缺失。

### 1.2 当前真实故障点

GMAC 的问题不是 probe 阶段，而是 **eth0 open 阶段**：

```text
rk_gmac-dwmac fe300000.ethernet eth0: Failed to reset the dma
rk_gmac-dwmac fe300000.ethernet eth0: stmmac_hw_setup: DMA engine initialization failed
rk_gmac-dwmac fe300000.ethernet eth0: __stmmac_open: Hw setup failed
```

也就是说：

- GMAC 控制器能被识别
- RGMII 模式能初始化
- PHY 能 attach
- `eth0` 能出现
- 但 DMA software reset 无法完成

### 1.3 当前更高概率根因

经过多轮 DTB 测试后，当前最高概率根因已经收敛为：

> **主线/ophub 6.18 的 `dwmac-rockchip` 在 RK3399Pro 上的 clock / GRF / reset / DMA init 顺序，与 vendor 4.4 驱动存在关键差异。**

换句话说：

- 纯 DTB 调参大概率已经接近上限；
- 下一步最该做的是 **驱动侧比对与最小补丁**，而不是继续盲改 delay / reset GPIO / 时钟方向。

## 2. 当前可用的保守基线

### 2.1 当前建议保留的测试启动项

当前已恢复到相对保守、信息量更高的测试 DTB：

```text
/mnt/sdboot/dtb/rockchip/gmac-phyhandle-pll-test.dtb
```

`extlinux.conf` 当前应指向：

```text
FDT /dtb/rockchip/gmac-phyhandle-pll-test.dtb
```

### 2.2 这个 DTB 的价值

它同时具备：

- vendor 风格 `vcc_phy`
- 显式 `phy-handle`
- 只绑定 `ethernet-phy@0`
- 抑制原先 `stmmac-0:00` / `stmmac-0:03` 双地址现象
- 启动稳定，不会把系统搞挂

所以它适合作为 **后续驱动补丁验证基线**。

---

## 3. 已经验证过、基本可以停止重复尝试的 DTB 方向

以下方向都已经测过，且 **都不能解决 DMA reset failed**：

1. 仅把 `/ethernet@fe300000/status = "okay"`
2. `rx_delay = 0x20`
3. `snps,reset-gpio = PC0`
4. `clock_in_out = output`（旧版不一致测试）
5. vendor neardi 延时参数：
   - `clock_in_out = input`
   - `PB7`
   - `tx_delay = 0x21`
   - `rx_delay = 0x15`
6. vendor 风格 `vcc_phy/phy-supply`
7. 显式 `phy-handle + mdio/ethernet-phy@0`
8. 一致的 `clock_in_out = output` + 删除 `assigned-clocks/assigned-clock-parents`

### 3.1 这些测试带来的真实收敛

已基本排除：

- 只是 delay 值不对
- 只是 reset GPIO 不对
- 只是 `phy-supply` 不对
- 只是双 PHY 地址扫描干扰
- 只是 input/output 方向写错

所以 **不要再继续盲试 delay / GPIO / status 组合**，投入产出已经很低。

---

## 4. 当前最关键的新证据

### 4.1 `ethtool -d eth0` 前后 DMA 寄存器几乎不变

在一致 output 测试下，抓到了非常关键的现象：

- `ip link set eth0 up` 前后，DMA dump 基本不变
- 例如：

```text
DMA Reg0  = 0x00020101
DMA Reg10 = 0x00110001
DMA Reg22 = 0x000D0F17
```

这个现象很重要，说明：

> 驱动在 open 阶段并没有把 DMA 控制器推进到“真正发生状态切换”的阶段。

这更像：

- reset 没真正生效
- 某个前置 clock 没真正被打通
- GRF/RGMII mode 配置没真正落到硬件期望状态
- 或 RK3399 路径上缺少 vendor 私有初始化步骤

### 4.2 显式 PHY 节点后，只剩一个 MDIO 地址

显式 `phy-handle` 后，MDIO 总线只剩：

```text
stmmac-0:00
```

不再出现：

```text
stmmac-0:03
```

但 DMA 问题依旧。

这说明：

> 双 PHY 地址现象不是当前 DMA 失败的直接根因。

### 4.3 output 模式下 `clk_mac_refout` 仍未真正启用

一致 output 测试下，运行时 `clk_summary` 仍显示：

```text
clk_mac_ref     disabled
clk_mac_refout  disabled
```

这提示：

- 即使 DTB 写成 output，runtime 参考时钟路径也未必真正切换；
- 很可能要靠驱动显式处理，而不是只靠 DT 属性。

## 5. 后续 agent 最应该做的工作顺序（强烈建议按这个顺序）

## Path A：驱动侧最小补丁（最高优先级）

### A1. 先拿到当前 6.18 对应源码

目标不是泛泛找源码，而是要拿到 **当前 SD 卡运行这个 6.18.33 镜像对应的 `dwmac-rk.c` / `stmmac` 源码**。

重点路径：

- 当前运行内核来源：`ophub rk35xx 6.18.33`
- 本地已有线索：
  - `lpa3399pro-armbian/build-armbian/kernel/rk35xx/6.18.33/*.tar.gz`
  - `lpa3399pro-armbian/rebuild`
  - 以及 ophub/kernel 构建逻辑

如果本地没有完整源码，下一步 agent 应直接从对应构建来源取到：

- `drivers/net/ethernet/stmicro/stmmac/dwmac-rk.c`
- `drivers/net/ethernet/stmicro/stmmac/dwmac1000_dma.c`
- `drivers/net/ethernet/stmicro/stmmac/stmmac_main.c`
- `drivers/net/phy/motorcomm.c`

### A2. 与 vendor 4.4 做针对性 diff

vendor 参考文件：

```text
LPA3399Pro-SDK-Linux-V3.0/kernel/drivers/net/ethernet/stmicro/stmmac/dwmac-rk.c
```

建议重点比对 RK3399 路径上的以下内容：

1. `rk3399_set_to_rgmii()` 或等价函数
2. GRF 写入顺序
3. tx/rx delay bitfield 写法
4. `clock_in_out` 对应的 ref/refout 时钟路径
5. `gmac_clk_enable()` / `clk_prepare_enable()` 顺序
6. reset assert/deassert 顺序
7. PHY reset GPIO 触发时序
8. DMA reset 前是否有额外 delay / barrier / readback
9. open 阶段是否有 vendor 私有 workaround

### A3. 第一批最值得尝试的最小补丁

建议优先做“小改动、高信息量”的补丁，而不是重写整个驱动。

#### 补丁候选 1：补齐 vendor RK3399 的 GRF/clock init 顺序

目标：

- 把 vendor 里 RK3399 相关的 mode/clock/GRF 初始化顺序，尽量原样移植到 6.18 `dwmac-rk.c`
- 尤其是 `set_to_rgmii()` 前后顺序

#### 补丁候选 2：在 open 前强制重新做 GMAC reset + 小延时

目标：

- 在 `stmmac_hw_setup()` 之前
- 对 `stmmaceth` reset 做一次明确的 assert/deassert
- 并插入 `usleep_range()` / `mdelay()`

因为当前现象很像 reset 根本没真正到位。

#### 补丁候选 3：output 模式下显式启用 `clk_mac_refout`

当前 DT 写 output 后，runtime 里 `clk_mac_refout` 还是 disabled。

所以如果 driver 里没有显式 enable refout，应该补：

- `clk_prepare_enable(clk_mac_refout)`
- 或明确 parent select
- 并打印 debug 日志确认实际走到了哪个分支

#### 补丁候选 4：DMA soft reset 前加寄存器读回/轮询前置检查

因为 `ethtool -d` 表现出 DMA 状态几乎没变，建议在驱动里加 debug：

- 打印 reset 前后的 DMA bus mode
- 打印 SWR bit 写入前后值
- 打印 timeout 时的寄存器快照

这个补丁即使不修复，也能极大提升后续定位效率。

## Path B：做“驱动补丁前的低风险运行时验证”

如果后续 agent 还没准备好编译，可先做这几件运行时验证，成本低、收益高。

### B1. 保留当前 `gmac-phyhandle-pll-test.dtb`

不要再换成更激进的 DTB，避免把问题复杂化。

### B2. 继续抓以下运行态信息

建议固定抓：

```bash
dmesg | grep -iE 'gmac|stmmac|eth0|yt8521|dma'
grep -E '(gmac|clkin_gmac|clk_mac_ref|clk_mac_refout|mac_clk_rx|mac_clk_tx)' /sys/kernel/debug/clk/clk_summary
ethtool -d eth0
ip link set eth0 up
ethtool -d eth0
```

### B3. 如果能用 driver debug/dynamic debug，优先加日志，不要再堆 DTB

比起继续改 DTB，更推荐：

- 给 `dwmac-rk.c` / `stmmac_main.c` 加 `dev_info()`
- 打印 open 阶段 clock/reset/寄存器状态

---

## Path C：如果 6.18 驱动补丁推进受阻，再考虑替代内核路径

这是备选，不是第一优先级。

### C1. vendor 4.4 内核路径

如果目标是“先让有线网卡工作”，而不是坚持主线 6.18，那么 vendor 4.4 是最有机会快速成功的路径。

但代价是：

- 老内核
- 兼容性/维护性差
- 后续主线合并价值低

### C2. 寻找更接近 RK3399Pro 的 6.1 / 6.6 / 6.18 已知可用树

如果能找到：

- 已经针对 RK3399Pro / TB-RK3399ProD / LPA3399Pro 做过 dwmac 修复的 tree
- 或更接近 vendor 的 RK3399 mainline patch set

那么比自己纯手搓更快。

但这属于“备援方案”，仍低于直接修当前 6.18 驱动。
