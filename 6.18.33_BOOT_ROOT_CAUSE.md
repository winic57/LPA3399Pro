# 6.18.33 SD 卡启动失败根因分析与修复

> 日期: 2026-06-17 21:30
> 参考: LPA3399Pro_SDCard_Full_Adaptation_Log_20260615.md

## 根因分析

### boot.scr 乱码的真正原因

之前误判为 boot.scr 文件损坏，实际上文件完全正确（与原始镜像 MD5 一致）。真正原因是：

**vendor U-Boot (SDK 版本) 不兼容 Armbian 的 boot.scr 脚本格式。**

证据链：

1. **U-Boot 缺少 DTB 支持**（日志 Line 155-156）:
   ```
   No resource partition
   Failed to load DTB
   Failed to get kernel dtb, ret=-19
   ```
   U-Boot 无法加载自身的 DTB，导致 `fdt` 命令不可用。

2. **环境变量损坏**（日志 Line 158）:
   ```
   *** Warning - bad CRC, using default environment
   ```
   U-Boot 使用默认环境，`${kernel_addr_r}` 等变量可能不正确。

3. **boot.scr 执行乱码**（日志 Line 203-204）:
   ```
   ## Executing script at 00500000
   Unknown command '��������...'
   SCRIPT FAILED: continuing...
   ```
   boot.scr 脚本中引用了 `${devtype}`, `${devnum}`, `${prefix}` 等变量，这些变量在默认环境中未定义或值不正确，导致脚本内容被错误解析。

4. **不扫描 extlinux**（日志 Line 198-206）:
   ```
   mmc1 is current device
   Scanning mmc 1:1...
   Found U-Boot script /boot.scr    ← 找到 boot.scr 就执行
   ...SCRIPT FAILED...               ← 失败后直接跳到 mmc0
   switch to partitions #0, OK
   mmc0(part 0) is current device    ← 没有在 mmc1 上尝试 extlinux！
   ```
   U-Boot 的 distro_bootcmd 中，boot.scr 优先级高于 extlinux。boot.scr 存在时不会尝试 extlinux。

### ophub 构建的三个缺陷

| # | 缺陷 | 说明 |
|---|---|---|
| 1 | `extlinux.conf` 被命名为 `.bak` | ophub 构建脚本将 extlinux.conf 重命名为 extlinux.conf.bak，导致 U-Boot 找不到 |
| 2 | `uInitrd` 目标文件缺失 | 软链接 `uInitrd → uInitrd-6.18.33-rk35xx-ophub` 指向不存在的文件 |
| 3 | 只有 boot.scr 可用 | vendor U-Boot 解析 boot.scr 时因环境变量缺失而乱码 |

### 对比 6.1.141 成功配置

6.1.141 版本（iter6 到 iter45）能正常启动的原因：

1. **iter6 镜像基于 hybrid_sdkboot.img** — 该镜像将完整引导链（idbloader + uboot + trust）替换为 SDK 版本
2. **使用 extlinux.conf 启动** — 不是 boot.scr（适配日志 §4.1 明确使用 extlinux.conf）
3. **DTB 经过 37+ 轮迭代调整** — 禁用 display-subsystem、sdhci、rkisp1 等挂死源

## 修复方案

### 修复内容

| # | 操作 | 原因 |
|---|---|---|
| 1 | 禁用 boot.scr（重命名为 .disabled） | 避免 vendor U-Boot 执行乱码脚本 |
| 2 | 创建 extlinux.conf | U-Boot distro_bootcmd 支持 extlinux 格式 |
| 3 | 不使用 INITRD | uInitrd 目标文件不存在，6.18.33 内核可以无 initrd 启动 |
| 4 | 参考 6.1.141 成功参数 | console, rootwait, rootdelay 等 |

### extlinux.conf 内容

```
LABEL Armbian
  LINUX /Image
  FDT /dtb/rockchip/rk3399pro-neardi-linux-lc110-base.dtb
  APPEND root=UUID=d52f7f8b-fa0e-4c9f-a552-d6df4329ac87 rootflags=data=writeback rw rootwait rootdelay=5 rootfstype=ext4 console=ttyS2,1500000 console=tty1 panic=0 usbcore.autosuspend=-1 initcall_blacklist=psci_checker initcall_debug printk.devkmsg=on earlycon=uart8250,mmio32,0xff1a0000,1500000n8 net.ifnames=0
```

### 关键参数说明（参考适配日志 §4.1）

| 参数 | 作用 | 来源 |
|---|---|---|
| `root=UUID=d52f7f8b...` | rootfs 分区 UUID | `blkid /dev/sdc2` 确认 |
| `console=ttyS2,1500000` | TTL 串口控制台 | 适配日志验证 |
| `console=tty1` | HDMI 控制台 | 适配日志验证 |
| `rootwait rootdelay=5` | 等待 SD 卡就绪 | 适配日志验证 |
| `panic=0` | 不自动重启 | 适配日志验证 |
| `usbcore.autosuspend=-1` | USB 稳定性 | 适配日志验证 |
| `initcall_blacklist=psci_checker` | 避免 psci 死锁 | 适配日志 §2 |
| `initcall_debug printk.devkmsg=on` | 详细启动日志 | 适配日志验证 |
| `earlycon=uart8250,...` | 早期控制台 | 适配日志验证 |

## 潜在风险

### 1. 缺少 initrd

uInitrd 文件缺失，如果 6.18.33 内核需要 initrd 来加载某些驱动（如文件系统模块），可能导致启动失败。

**回退方案**：如果启动失败提示找不到 rootfs，需要在 rootfs 分区的 `/boot/` 中创建 initrd。

### 2. DTB 未经过适配

6.18.33 镜像的 DTB 是 ophub 默认的，没有经过适配日志中的 37 轮迭代调整。可能导致：
- display-subsystem 死锁（iter7 修复）
- sdhci eMMC 探测死锁（适配日志 §4.2）
- rkisp1/mipi-dphy 死锁
- iep 硬挂死

**回退方案**：如果启动卡死在某个驱动，需要参考适配日志 §4.2 用 `fdtput` 禁用相应节点。

### 3. U-Boot 环境变量

`bad CRC, using default environment` 意味着 U-Boot 每次启动都用默认环境。extlinux 不依赖环境变量，所以不影响启动。但如果需要持久化其他配置，需要 `saveenv`。
