这进一步确认：

## 3. 失败点

系统启动到 userspace 后，网络服务或 udev/network 管理尝试拉起 eth0，随后失败：

```text
rk_gmac-dwmac fe300000.ethernet eth0: Register MEM_TYPE_PAGE_POOL RxQ-0
rk_gmac-dwmac fe300000.ethernet eth0: PHY [stmmac-0:00] driver [YT8521 Gigabit Ethernet] (irq=POLL)
rk_gmac-dwmac fe300000.ethernet eth0: Failed to reset the dma
rk_gmac-dwmac fe300000.ethernet eth0: stmmac_hw_setup: DMA engine initialization failed
rk_gmac-dwmac fe300000.ethernet eth0: __stmmac_open: Hw setup failed
```

## 4. 当前判断

本次测试证明“只打开 GMAC status”不会立刻导致整板卡死，但 eth0 无法正常 up。

当前真正卡点是：

```text
__stmmac_open -> stmmac_hw_setup -> DMA reset failed
```

这通常更偏向以下方向：

1. MAC/GMAC 时钟输入不对或没有稳定；
2. reset GPIO / PHY reset 时序不匹配；
3. RGMII clock parent / `clock_in_out` 方向不匹配；
4. GRF/RGMII 配置与当前板级硬件不一致；
5. 主线 `dwmac-rk.c` 与 vendor 4.4 对 RK3399Pro/YT8521 的初始化差异。

`rx_delay` 主要影响链路收发时序，一般不是 DMA soft reset 失败的首要原因，但仍可作为后续独立变量测试。

## 5. 下一步建议

### 建议 1：先禁止自动拉起 eth0，保留 GMAC probe

目的：让系统启动后停在 eth0 存在但 down 的状态，方便 TTL 手动检查。

可在 bootargs 临时加入：

```text
systemd.mask=NetworkManager.service systemd.mask=systemd-networkd.service
```

或离线屏蔽 NetworkManager 相关服务。

启动后检查：

```bash
ip link
cat /sys/class/net/eth0/operstate
ethtool -i eth0
ethtool eth0
find /sys/bus/mdio_bus/devices -maxdepth 2 -type f -name phy_id -print -exec cat {} \;
dmesg | grep -iE 'gmac|stmmac|dwmac|eth0|phy|yt8521|motorcomm'
```

### 建议 2：下一轮 DTB 变量优先级

建议不要一次改多个变量，按顺序制作独立测试 DTB：

1. `gmac-test.dtb`: status=okay，保持 PB7 + rx_delay 0x11；本轮已完成，结果为 DMA reset failed。
2. `gmac-rx20-test.dtb`: 仅改 `rx_delay = <0x20>`。
3. `gmac-pc0-test.dtb`: 在确认板级连线可能是 Toybrick prod 后，单独测试 reset GPIO 从 PB7 改 PC0。
4. `gmac-clockout-test.dtb`: 单独测试 `clock_in_out = "output"` 或 clock parent 变体。
5. 若仍失败，进入驱动侧对比 vendor 4.4 `dwmac-rk.c` 与当前 6.18 主线逻辑。

## 6. 本轮结论

GMAC enable 后：

- ✅ 内核能加载 `gmac-test.dtb`
- ✅ GMAC driver probe 成功
- ✅ YT8521 PHY 被识别

所以后续修复重点不是“补驱动”，而是 GMAC 时钟/reset/GRF/PHY 初始化路径。
