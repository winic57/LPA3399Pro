# 6.18.33 GMAC 显式 PHY 子节点 + keep-pll 测试结果（2026-06-18）

## 1. 目的

在 `vcc_phy` 已补齐但 DMA reset failed 仍存在的前提下，进一步验证：

1. 当前默认 MDIO 自动扫描挂出两个 PHY 地址（`stmmac-0:00` / `stmmac-0:03`）是否会干扰主 PHY 选择；
2. YT8521 是否需要通过显式 PHY 节点传递额外属性，例如保持 PLL 输出、禁止 auto sleep。

## 2. 变更

基于：

```text
/mnt/sdboot/dtb/rockchip/gmac-vccphy-test.dtb
```

创建：

```text
/mnt/sdboot/dtb/rockchip/gmac-phyhandle-pll-test.dtb
```

新增内容：

```dts
phy-handle = <&phy0>;

mdio {
    compatible = "snps,dwmac-mdio";
    #address-cells = <1>;
    #size-cells = <0>;

    ethernet-phy@0 {
        reg = <0>;
        motorcomm,keep-pll-enabled;
        motorcomm,auto-sleep-disabled;
    };
};
```

等效目标：

- 显式指定只使用 PHY 地址 `0x00`；
- 不再依赖“Deprecated MDIO bus assumption used”的默认扫描逻辑；
- 尝试让 YT8521 保持 PLL/时钟输出。

## 3. 运行态验证

运行态 DT 确认：

```text
/proc/device-tree/ethernet@fe300000/phy-handle = <0xc2>
/proc/device-tree/ethernet@fe300000/mdio/ethernet-phy@0  存在
```

MDIO 总线结果由原先的两个地址：

```text
stmmac-0:00
stmmac-0:03
```

收敛为仅一个地址：

```text
stmmac-0:00
```

这说明显式 `phy-handle + mdio/ethernet-phy@0` 生效，双地址现象已被抑制。

## 4. dmesg 结果

GMAC 依然 probe 正常，并且只绑定到 `stmmac-0:00`：

```text
rk_gmac-dwmac fe300000.ethernet: clock input or output? (input).
rk_gmac-dwmac fe300000.ethernet: TX delay(0x21).
rk_gmac-dwmac fe300000.ethernet: RX delay(0x15).
rk_gmac-dwmac fe300000.ethernet: init for RGMII
rk_gmac-dwmac fe300000.ethernet eth0: PHY [stmmac-0:00] driver [YT8521 Gigabit Ethernet] (irq=POLL)
```

但 `eth0` open 仍失败：

```text
rk_gmac-dwmac fe300000.ethernet eth0: Failed to reset the dma
rk_gmac-dwmac fe300000.ethernet eth0: stmmac_hw_setup: DMA engine initialization failed
rk_gmac-dwmac fe300000.ethernet eth0: __stmmac_open: Hw setup failed
```

## 5. 结论

**结论：显式 PHY 子节点、固定 `phy-handle` 到 `0x00`、并尝试 `motorcomm,keep-pll-enabled` / `motorcomm,auto-sleep-disabled` 后，仍不能解决 DMA reset failed。**

但这轮结果非常关键，因为它把问题进一步收敛了：

1. 双 PHY 地址（`0x00` / `0x03`）不是 DMA reset failed 的直接根因；
2. 至少在当前 6.18 驱动路径下，增加这些 PHY 侧属性也没有改变 DMA reset 行为；
3. 主嫌疑继续向 **MAC/CRU/GRF/reset/init sequence** 偏移。

## 6. 对后续分析的意义

本轮已经把“PHY 绑定方式不对”“MDIO 扫描错误选到错误 PHY”“YT8521 需要简单 PLL 属性即可恢复”这些方向基本排除。

因此下一步更值得尝试的是：

- 统一的 `clock_in_out=output` 测试；
- 或直接转入 `dwmac-rockchip`/`stmmac` 驱动侧差异分析。
