## 二、详细修改记录

### 修改 1: idbloader 替换 (2026-06-17 20:30)

**问题**: ophub 默认 idbloader 无法被 RK3399 BootROM 识别，板子跳过 SD 卡从 eMMC 启动 4.4.194。

**操作**:
```bash
sudo dd if=/mnt/sdb3/LPA3399Pro/LPA3399Pro-SDK-Linux-V3.0/idbloader.img \
        of=/dev/sdc seek=64 conv=notrunc,fsync
```

| 项目 | 值 |
|---|---|
| 源文件 | `/mnt/sdb3/LPA3399Pro/LPA3399Pro-SDK-Linux-V3.0/idbloader.img` |
| 大小 | 203036 字节 (198 KiB) |
| 写入位置 | sector 64 (偏移 32KB) |
| MD5 | `3fa843da66820d758f6000266af7934f` |

**效果**: BootROM 输出 `Found IDB in SDcard`，从 SD 卡启动 U-Boot。

### 修改 2: 禁用 boot.scr，恢复 extlinux.conf (2026-06-17 21:28)

**问题**: vendor U-Boot (SDK 版本) 解析 boot.scr 时输出乱码，脚本执行失败。根因：
1. U-Boot 无法加载自身 DTB → `Failed to load DTB`
2. 环境变量 CRC 校验失败 → `bad CRC, using default environment`
3. 默认环境中 `${devtype}` 等变量不正确，导致 boot.scr 脚本解析异常

**ophub 构建缺陷**:
- `extlinux.conf` 被错误命名为 `extlinux.conf.bak`
- `uInitrd` 软链接指向不存在的文件

**操作**:
```bash
# 禁用 boot.scr
mv boot.scr boot.scr.disabled

# 创建 extlinux.conf
cat > extlinux/extlinux.conf << 'EOF'
LABEL Armbian
  LINUX /Image
  FDT /dtb/rockchip/rk3399pro-neardi-linux-lc110-base.dtb
  APPEND root=UUID=d52f7f8b-fa0e-4c9f-a552-d6df4329ac87 rootflags=data=writeback rw rootwait rootdelay=5 rootfstype=ext4 console=ttyS2,1500000 console=tty1 panic=0 usbcore.autosuspend=-1 initcall_blacklist=psci_checker initcall_debug printk.devkmsg=on net.ifnames=0
EOF
```

**效果**: U-Boot 找到 `/extlinux/extlinux.conf`，正确加载内核 Image 和 DTB。

### 修改 3: 替换为主线 DTB (2026-06-18 07:15)

**问题**: vendor DTB 的时钟绑定格式与主线内核 6.18.33 不兼容：
```
clk: couldn't get clock 5 for /clock-controller@ff760000
rockchip_clk_of_add_provider: could not register clk provider
```

vendor DTB `/chosen` 节点还包含 embedded bootargs，覆盖了 extlinux.conf 的参数：
- `console=ttyFIQ0`（vendor 专用，主线内核不存在）替换了 `console=ttyS2,1500000`
- `root=PARTUUID=614e0000-0000` 替换了正确的 UUID

**操作**:
```bash
# 备份 vendor DTB
cp rk3399pro-neardi-linux-lc110-base.dtb rk3399pro-neardi-linux-lc110-base.dtb.vendor.bak

# 替换为主线 DTB
cp rk3399pro-rock-pi-n10.dtb rk3399pro-neardi-linux-lc110-base.dtb
```

| 项目 | 值 |
|---|---|
| 源文件 | `dtb-rockchip-6.18.33-rk35xx-ophub.tar.gz` 内的 `rk3399pro-rock-pi-n10.dtb` |
| 源路径 | `/mnt/sdb3/LPA3399Pro/lpa3399pro-armbian/build-armbian/kernel/rk35xx/6.18.33/dtb-rockchip-6.18.33-rk35xx-ophub.tar.gz` |
| 大小 | 59K (vendor DTB 为 100K) |
| compatible | `radxa,rockpi-n10`, `vamrs,rk3399pro-vmarc-som`, `rockchip,rk3399pro` |
| chosen | `stdout-path = "serial2:1500000n8"` (无 embedded bootargs) |

**主线 DTB GMAC 节点配置**:
```
compatible = "rockchip,rk3399-gmac"
phy-mode = "rgmii"
clock_in_out = "input"
tx_delay = <0x28>
rx_delay = <0x11>
snps,txpbl = <0x04>
status = "okay"
```

**效果**: 时钟驱动正常注册，串口控制台 `ttyS2` 正确启用，bootargs 不再被覆盖。

### 修改 4: 禁用 DRM/display/watchdog (2026-06-18 07:25)

**问题**: 内核在 VOP (Video Output Processor) 加入 iommu group 后卡死（0.649s）。参考适配日志 iter7 经验：display-subsystem 初始化死锁。

**操作** (fdtput):
```bash
fdtput -t s $DTB /display-subsystem status disabled
fdtput -t s $DTB /vop@ff8f0000 status disabled
fdtput -t s $DTB /vop@ff900000 status disabled
fdtput -t s $DTB /watchdog@ff848000 status disabled
```

**效果**: 实际未改变卡死位置（卡死点在日志缓冲区溢出，非 DRM）。

**问题**: `initcall_debug` 产生大量日志，导致 952 条 printk 消息被丢弃，看不到后续驱动加载日志。

### 修改 6: 禁用 GMAC + PCIe (2026-06-18 07:40)

**问题**: 日志在 YT8521 PHY attach（1.529s）后停止，串口无响应，板子死机。

**根因**: 适配日志 iter45 已确认 GMAC DMA 硬件缺陷：
- `DMA Bus_Mode SWR 位永久卡死 = 1`
- 写入 0 后仍读回 1，硬件不可清除
- 主线 stmmac 驱动在 probe 阶段自动执行 DMA reset → 整板挂死

（vendor 4.4.194 内核只在 `ip link set eth0 up` 时触发 DMA reset，probe 阶段不触发，所以不会自动死机）

**操作** (fdtput):
```bash
fdtput -t s $DTB /ethernet@fe300000 status disabled
fdtput -t s $DTB /pcie@f8000000 status disabled
```

**预期效果**: 绕过 GMAC DMA 死锁，系统应能继续启动到 rootfs 挂载和用户空间。

## 八、修改 7 分析: GMAC+PCIe 禁用后的启动回归 (2026-06-18 08:30)

### 8.1 新增 TTL 日志

修改 6 完成后产生了两份新日志:

| 日志文件 | 大小 | 时间 | DTB 大小 | GMAC | PCIe | 最远进度 |
|---|---|---|---|---|---|---|
| `6.18.33_gmac_disabled_20260618_074651.log` | 41K | 06-18 07:46 | 60186B | 禁用 | 禁用 | 0.681s |
| `6.18.33_drm_disable_20260618_081210.log` | 12K | 06-18 08:12 | 60186B | 禁用 | 禁用 | 0.684s |

两份日志使用**同一 DTB** (60186 字节)，配置完全相同。

### 8.2 回归现象

**修改 6 (GMAC + PCIe 同时禁用) 导致系统挂死点提前**:

| 配置 | 最远进度 | fe320000.mmc | 挂死位置 |
|---|---|---|---|
| drm_disable_073548 (PCIe✓ GMAC✓) | **1.529s** | ✅ 始终成功 (0.698s) | GMAC YT8521 PHY attach |
| gmac_disabled_074651 boot1 (PCIe✗ GMAC✗) | **0.665s** | — | cpu4 EM 创建后，DDR 复位 |
| gmac_disabled_074651 boot2 (PCIe✗ GMAC✗) | **0.681s** | ❌ Version ID 读取截断 | fe320000.mmc "Ver..." |
| drm_disable_081210 (PCIe✗ GMAC✗) | **0.684s** | ✅ 偶然成功 (Version ID = 270a) | fe310000.mmc 探测后日志截断 |

> **结论**: 禁用 PCIe 导致系统从 1.529s 回归到 0.665~0.684s，**提前约 0.85 秒挂死**。

### 8.3 根因分析

#### 直接原因: fe320000.mmc (SDIO/WiFi) 探测竞态

1. **gmac_disabled_074651** (两次启动均失败):
   - Boot 1: 挂死在 0.665s (cpu4 EM 之后)，未到达 fe320000.mmc
   - Boot 2: 挂死在 fe320000.mmc Version ID 寄存器读取 (0.681s，消息截断为 "Ver")
   - → Version ID 读取 (第一次寄存器访问) 即挂死，说明总线访问 stall

2. **drm_disable_081210** (fe320000 偶然成功):
   - fe320000.mmc Version ID = 270a (成功)，但日志在 fe310000.mmc 探测后 (0.684s) 截断
   - → 同一 DTB 下结果不一致，证明问题是**间歇性**的 (时序依赖)

3. **drm_disable_073548** (PCIe 启用，fe320000 始终成功):
   - fe320000.mmc 在 0.698s 成功探测 (Version ID = 270a，irq 86)
   - fe310000.mmc 在 0.701s 成功重探测 (Version ID = 270a，irq 48)
   - → PCIe 启用时 fe320000 从不失败

#### 时序窗口假说

PCIe 驱动 link training 过程持续约 0.7 秒 (drm_disable_073548 日志: 0.699s 启动 → 1.412s 超时)。

| 时序 | PCIe 启用时 | PCIe 禁用时 |
|---|---|---|
| 0.65s | cpu4 EM 创建 | cpu4 EM 创建 |
| 0.68s | — | **fe320000/fe310000 竞态触发** |
| 0.70s | PCIe 开始 link training (占用 CPU/总线) | fe320000.mmc 探测完成 |
| 0.77s | SD 卡检测完成 | — |
| 1.41s | PCIe link training 超时 (-110) | — |

PCIe link training 期间占用 PCIe 控制器寄存器空间和部分总线带宽，**无意中为 dwmmc 驱动的并行探测提供了时序隔离**。禁用 PCIe 后启动变快，fe320000.mmc 和 fe310000.mmc 几乎同时探测，暴露了竞态。

#### 关键证据

1. **PCIe 超时是优雅失败**: drm_disable_073548 日志第 578-579 行确认 PCIe 以 -110 错误退出后系统继续正常启动到 GMAC 探测
2. **fe320000 无 WiFi 子节点**: 主线 DTB 中 `/mmc@fe320000` 无子节点 (已通过 `fdtget -l` 确认)，仅有 `cd-gpios` 属性，WiFi 在本板上不存在

   - `/ethernet@fe300000` (GMAC): status="okay", compatible="rockchip,rk3399-gmac" ✅
   - `/pcie@f8000000`: status="okay" ✅
   - `/pcie-ep@f8000000`: status="disabled" (已是禁用状态)
   - `/mmc@fe320000`: status="okay", 有 `power-domains`、`cd-gpios` 属性 ✅

### 8.4 修改 6 评估

| 节点 | 禁用原因 | 是否必要 | 评估 |
|---|---|---|---|
| `/ethernet@fe300000` (GMAC) | iter45 确认 DMA SWR 硬件缺陷 | ✅ **必要** | 主线 stmmac probe 阶段触发 DMA reset → 整板挂死 |

## 十二、TB-RK3399ProD 参考对 GMAC 修复的补充结论 (2026-06-18)

已通过代理访问并本地克隆 `https://github.com/yifengyou/TB-RK3399ProD`，本地仓库 HEAD 为 `2fb376c`。未发现可直接替换当前 SD 启动环境的已编译 `.dtb`，主要可用内容是 DTS/DTSI 和 6.x patch 参考。可用参考如下：

- `docs/设备树适配/设备树适配.md`：GMAC 建议为 `status = "okay"`，`snps,reset-gpio = <&gpio3 RK_PC0 GPIO_ACTIVE_LOW>`，`rx_delay = <0x20>`。
- `kernel-4.4.194-official/dts/rk3399pro-toybrick-prod.dtsi`：同样是 `PC0 + rx_delay 0x20`。
- `kernel-4.4.194-official/dts/rk3399pro-toybrick-96ai.dtsi`：`PC0 + rx_delay 0x11`。
- `kernel-4.4.194-official/dts/rk3399pro-evb-v11-linux.dts`：`PB7 + rx_delay 0x11`。
- `ophub_6.6.y/0001-add-toybrick-rk3399prod-support-for-kernel-6.patch`：6.x 适配仍采用 `PC0 + rx_delay 0x20`，并将 GMAC 设为 `okay`。

### 结论

当前板子的有线网卡问题**不是内核缺少 `dwmac-rockchip` 驱动**，而是当前稳定 DTB 中 `/ethernet@fe300000` 被禁用，且历史测试显示 GMAC probe/open 可能触发死机。

### 下一步建议

1. 先保留当前稳定基线不动。
2. 复制出独立测试 DTB，只把 `/ethernet@fe300000/status` 改成 `okay`。
3. 若能 probe，再单独试 `rx_delay=0x20`。
4. 只有在确认板级连线一致时，再尝试把 `snps,reset-gpio` 从 PB7 改成 PC0。
5. 若 probe/open 仍死机，则回到 `dwmac-rk.c`/PHY reset/clock 逻辑继续排查。
